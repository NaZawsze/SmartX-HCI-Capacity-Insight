# v0.5.1u2 过渡版本说明

发布日期：2026-06-28

## 版本定位

`v0.5.1u2` 是平台过渡版本，用于把现有 v2 环境整理到统一部署和升级口径，为后续单独升级 `upgrade-runner v0.3.1` 做准备。

支持升级范围：

- `v0.5.0 -> v0.5.1u2`
- `v0.5.1 -> v0.5.1u2`
- `v0.5.1u1 -> v0.5.1u2`
- `v0.5.1u2 -> v0.5.1u2` 同版本重同步

不支持从 v1 或 `v0.4.x` 原地升级；旧架构系统继续通过“新装 v2 + 数据迁移包导入”兼容。

## 更新内容

- 平台版本、镜像 tag、README、部署文档、版本治理文档和升级包说明统一为 `v0.5.1u2`。
- 平台升级包说明写清兼容范围、必填文件、校验和、迁移边界和禁止包含的数据。
- `docker-compose.yml`、`docker-compose.release.yml`、`docker-compose.offline.yml` 保持固定顶层 `name`，避免用户直接执行 `docker compose up -d` 时生成错误项目名。
- 平台升级包继续只包含 `web-api`、`collector-worker`、`frontend` 三个业务服务镜像和白名单项目文件。
- 打包器默认最低来源版本为 `v0.5.0`，无 schema 迁移时不写 `migration`、`migration_steps` 或 `script.sandbox.v1`。

## 修复内容

- 修复部分环境因 compose 项目名不一致导致服务管理、平台升级和观测组件状态读取不稳定的问题。
- 修复升级包说明过轻，用户难以确认 sha256、schema、兼容范围和迁移边界的问题。
- 修复无 SQLite schema 变化的平台包误带迁移脚本或沙箱脚本能力要求的问题。

## Runner 边界

`v0.5.1u2` 不升级 `upgrade-runner`，也不包含 `upgrade-runner.tar`。当前 runner 仍按独立组件版本管理。

后续如果发布依赖新 runner 能力的平台版本，应先发布并升级 `upgrade-runner v0.3.1` 组件包；平台升级包必须在预检查阶段校验 runner 版本或 capability，不满足时拒绝升级并提示先升级 runner。

## 升级包内容

必填：

```text
manifest.json
checksums.sha256
release-notes.md
images/web-api.tar
images/collector-worker.tar
images/frontend.tar
project/**
```

可选：

```text
migrations/run_migrations.py
```

仅当来源版本到目标版本之间存在 SQLite schema 迁移时才包含。

禁止包含：

- `.env`
- SQLite 数据库
- Prometheus 历史数据 block
- Tower 账号、密码、token
- 运行时备份、导出文件或真实业务数据
- `upgrade-runner.tar`

## 发布检查

- `VERSION` 为 `v0.5.1u2`。
- `RUNNER_VERSION` 不随平台版本变更。
- 三个业务镜像 tag 为 `v0.5.1u2`。
- 升级包 manifest `version=v0.5.1u2`、`min_version=v0.5.0`、`schema_version=3`。
- 无迁移步骤时，升级包内没有 `migrations/run_migrations.py`，manifest 没有 `migration` 和 `migration_steps`。
